Intro & Basic Assumptions

Everybody has their own ideas about how a web services API should function. Here are the choices we made.

It's JSON. Everybody's good with this, right? Any XML fans here? ::crickets chirp::

It's RESTful, basically. Have you ever worked with SOAP web services? Those are dumb. HTTP itself is RESTful, so it makes the most sense to go with a web service architecture that takes advantage of that. Wikipedia has a quick overview and if you're a programmer this book is an excellent read. (It's from 2007, but is pretty much the definitive word on the matter...)

Authentication

Authentication is pretty essential to using the API. We've tried to make it really coders-friendly since duh, we're coders and we're eating our own dogfood here. Here's how authentication works...

  1. Get an authentication token. Post a request to http://api.otakubooty.com/api/Authentications with login and password parameters in the request body. The login parameter can be any login the member has ever used, past or present, if this a member that has used the "name change" feature.
  2. Hold on to the authentication token that was returned. The authenticationToken will be returned as part of the JSON response and as a cookie so if you're authenticating via Javascript in a browser you're all set.
  3. Include the authentication token in your next request! Again, if you authenticated via Javascript, the auth token should already be in your cookies.
